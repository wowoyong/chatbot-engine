# 챗봇 아키텍처 기초

## 최소 구성 요소

LLM 기반 챗봇은 최소 네 가지로 구성된다:

1. **LLM 클라이언트** — 모델 API 호출과 스트리밍 파싱. 구현체(Ollama, OpenAI 등)를 인터페이스 뒤에 숨기면 모델 교체가 쉬워진다.
2. **세션(대화 관리)** — 멀티턴 히스토리를 배열로 유지하고, 매 요청에 `[시스템 프롬프트, ...이전 대화, 새 질문]`으로 재조립해 전송한다.
3. **컨텍스트 관리자** — 히스토리가 컨텍스트 예산을 넘으면 오래된 것부터 잘라내거나(트리밍) LLM으로 압축(요약)한다.
4. **인터페이스** — CLI, 웹, API 등. 로직 없는 얇은 I/O 레이어로 유지해야 엔진을 여러 인터페이스에서 재사용할 수 있다.

## 스트리밍

"글자가 실시간으로 타이핑되는" 효과는 서버가 응답을 조각(청크) 단위로 흘려보내고 클라이언트가 도착 즉시 렌더링하는 것이다. 대표 전송 형식:

- **ndjson**: 한 줄에 JSON 하나씩 (Ollama native API). HTTP 청크는 라인 경계와 무관하게 잘리므로 수신 측에 라인 버퍼가 필수다.
- **SSE (Server-Sent Events)**: `data: ...\n\n` 형식의 표준 스트림 (OpenAI 호환 API, 웹 브라우저 EventSource). 재연결·이벤트 타입 등 규약이 있다.

멀티바이트 문자(한글 등)가 청크 경계에서 잘릴 수 있으므로 UTF-8 디코더는 스트리밍 모드로 써야 한다.

## 히스토리 관리 전략

- **쌍 단위 유지**: user/assistant 짝이 깨진 히스토리는 모델이 문맥을 오해한다. 트리밍은 쌍 단위로.
- **실패 시 미기록**: 스트림이 중간에 끊기면 반쪽 응답을 히스토리에 남기지 않는다 — 다음 턴의 컨텍스트 오염 방지.
- **원본 보존**: 트리밍·요약은 전송본에만 적용하고 원본 히스토리는 보존하면 언제든 재구성할 수 있다.

## 메모리(장기 기억) 전략

- **슬라이딩 윈도우**: 최근 N턴만 유지. 단순하지만 오래된 정보 소실.
- **요약 압축**: 잘려나갈 대화를 LLM으로 한 문단 요약해 시스템 메시지로 대체. "기억이 흐릿해지는" 자연스러운 동작. 요약 결과는 캐시해 재요약을 피한다.
- **영속화**: 세션을 파일/DB에 저장하면 프로세스 재시작 후에도 대화가 이어진다. 쓰기 도중 크래시 대비로 원자적 쓰기(임시 파일 → rename)를 쓴다.

## 우아한 성능 저하 (Graceful Degradation)

부가 기능의 실패가 핵심 기능(대화)을 멈추면 안 된다. 검색 실패 → 검색 없이 답변, 요약 실패 → 트리밍만, 저장 파일 손상 → 백업 후 새 세션. 유일하게 대화를 멈춰도 되는 경우는 LLM 서버 자체의 불능뿐이다.
