# LLM 기초

## LLM이란

대규모 언어 모델(Large Language Model)은 방대한 텍스트로 학습된 신경망으로, "지금까지의 텍스트 다음에 올 가장 그럴듯한 토큰"을 반복 예측하는 방식으로 동작한다. 챗봇의 답변은 이 다음-토큰 예측을 수백~수천 번 반복한 결과다. 모델은 학습 시점의 지식만 가지며(지식 컷오프), 실행 중에 스스로 학습하거나 기억을 갱신하지 않는다.

## 토큰

토큰은 모델이 텍스트를 처리하는 최소 단위다. 단어도 글자도 아닌 그 중간 단위로, 토크나이저가 결정한다. 영어는 대략 4글자당 1토큰, 한국어는 대략 1~2글자당 1토큰으로 영어보다 토큰 효율이 낮다. 같은 의미의 문장도 한국어가 더 많은 토큰을 소비하므로, 컨텍스트 예산 계산 시 언어별 차이를 고려해야 한다.

## 컨텍스트 윈도우

모델이 한 번에 볼 수 있는 토큰의 최대량이다. 프롬프트(시스템 지시 + 대화 이력 + 질문)와 생성될 응답이 모두 이 안에 들어가야 한다. 초과분은 조용히 잘리거나 오류가 나므로, 챗봇은 반드시 예산 관리(트리밍, 요약)를 해야 한다. 모델 스펙상 컨텍스트(예: qwen3의 40K)와 서빙 설정의 실제 컨텍스트(예: Ollama 기본 num_ctx 4096)는 다를 수 있다 — 실전에서는 서빙 설정이 진실이다.

## 샘플링 파라미터

- temperature: 다음 토큰 선택의 무작위성. 낮으면(0~0.3) 결정적·반복적, 높으면(0.8+) 다양하지만 산만해진다.
- top-p (nucleus sampling): 누적 확률 상위 p 안의 토큰만 후보로 남긴다. temperature와 함께 다양성을 조절한다.
- 코드 생성·사실 질의엔 낮은 temperature, 창작엔 높은 temperature가 일반적 권장이다.

## Thinking(추론) 모드

qwen3 같은 최신 모델은 답변 전에 내부 사고 과정(chain-of-thought)을 `<think>` 블록으로 생성할 수 있다. 복잡한 추론 품질은 오르지만 응답 지연과 토큰 소비가 커진다. 대화형 챗봇 UX에서는 보통 비활성(think: false)이 낫고, 수학·논리 문제에서만 켜는 것이 효율적이다.

## Stateless라는 본질

LLM API 호출은 상태가 없다. 모델은 이전 호출을 전혀 기억하지 못하며, "대화의 기억"처럼 보이는 것은 클라이언트가 매 요청마다 이전 대화 전체를 다시 보내기 때문이다. 챗봇 엔진의 핵심 책임 하나가 바로 이 히스토리 재조립이다.
