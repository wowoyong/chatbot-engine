# 로컬 LLM 서빙

## Ollama

로컬 머신에서 LLM을 HTTP 서버로 노출하는 가장 쉬운 도구. `ollama pull <모델>`로 받고 `ollama serve`가 `localhost:11434`에 API를 연다.

- **native API**: `/api/chat` (ndjson 스트리밍), `/api/embed` (배치 임베딩), `/api/tags` (설치 모델 목록)
- **OpenAI 호환 API**: `/v1/chat/completions` 등 — OpenAI SDK를 쓰는 도구를 base URL만 바꿔 로컬 모델에 연결할 수 있다. API 키는 형식상 아무 값이나 넣으면 된다.

## 양자화와 메모리 요구량

양자화(quantization)는 모델 가중치의 정밀도를 낮춰(예: FP16 → 4bit) 메모리를 줄이는 기법. Ollama 기본 배포는 대부분 Q4_K_M(4bit 계열)이다. 대략적인 상주 메모리:

| 모델 | 파라미터 | Q4 크기 | 실행 시 여유 포함 |
|------|---------|---------|-----------------|
| qwen3:8b | 8.2B | 약 5.2GB | 약 7~8GB |
| qwen3:14b | 14.8B | 약 9.3GB | 약 12~13GB |

**16GB 메모리 머신(예: 맥미니)에서는 14b 상주가 사실상 불가**하다 — OS와 다른 프로세스 몫을 빼면 여유가 없다. 8b가 현실적 상한이며, 이 프로젝트의 챗봇 런타임이 qwen3:8b로 고정된 이유다. 임베딩 모델(nomic-embed-text, 수백 MB)은 부담이 작다.

## 컨텍스트 길이와 메모리

컨텍스트(num_ctx)를 늘리면 KV 캐시 메모리가 선형 이상으로 증가한다. Ollama 기본은 4096이며, 모델 스펙(40K)까지 올리려면 그만큼 메모리가 더 필요하다. 예산이 빠듯한 머신에서는 기본값 유지가 안전하다.

## 모델 상주와 교체

Ollama는 요청이 오면 모델을 메모리에 올리고 유휴 시간이 지나면 내린다(keep_alive). 채팅 모델과 임베딩 모델을 번갈아 쓰면 로드/언로드가 반복될 수 있는데, 두 모델의 합이 메모리에 들어가면 동시 상주가 되어 빠르다. 8b(5GB) + 임베딩(0.3GB)은 16GB에서 동시 상주 가능하다.

## 서버 운영 팁

- 원격 접근이 필요하면 `OLLAMA_HOST=0.0.0.0`으로 바인딩을 바꾼다 (기본은 localhost 전용).
- 첫 토큰 지연(TTFT)은 모델 로드 + 프롬프트 처리 시간이다. 프롬프트(히스토리+발췌)가 길수록 첫 응답이 느려진다 — 컨텍스트 예산 관리가 곧 응답 속도 관리다.
