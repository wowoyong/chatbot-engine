# RAG (Retrieval-Augmented Generation) 기초

## 왜 RAG인가

LLM은 학습 시점 이후의 지식, 사내/개인 문서, 최신 정보를 모른다. RAG는 질문과 관련된 문서 조각을 검색해 프롬프트에 끼워 넣어, 모델이 "주어진 근거로 답하게" 만드는 기법이다. 파인튜닝 없이 지식을 갱신할 수 있고(문서만 바꾸면 됨), 출처 인용이 가능해진다.

## 파이프라인

```
[색인 시점]  문서 → 청킹 → 임베딩 → 벡터 인덱스 저장
[질의 시점]  질문 → 임베딩 → 유사도 검색(top-K) → 발췌를 프롬프트에 주입 → LLM 답변
```

## 임베딩

텍스트를 고정 차원의 숫자 벡터로 바꾸는 것. 의미가 비슷한 텍스트는 벡터 공간에서 가깝다. 전용 임베딩 모델(nomic-embed-text 등)을 쓰며, 채팅 모델과 별개다. 색인과 질의는 반드시 같은 임베딩 모델을 써야 한다 — 모델이 다르면 벡터 공간이 달라 유사도가 무의미해진다.

## 청킹

문서를 검색 단위로 자르는 것. 너무 크면 무관한 내용이 섞여 정밀도가 떨어지고, 너무 작으면 문맥이 부족하다. 실전 요령:

- 마크다운은 헤딩 단위로 먼저 자르고, 긴 섹션만 재분할한다.
- 재분할 시 겹침(overlap)을 둬 조각 경계의 문맥 단절을 줄인다.
- 헤딩 텍스트를 청크 임베딩 입력에 접두어로 포함하면 짧은 본문의 검색 품질이 오른다.

## 유사도 검색

코사인 유사도(두 벡터 사이 각도)가 표준이다. 소규모(수백~수천 청크)는 전수 비교로 충분하고, 대규모는 ANN 인덱스(HNSW 등)나 벡터 DB를 쓴다.

- **top-K**: 상위 K개만 주입 (보통 3~5). 많이 넣을수록 좋은 게 아니다 — 무관한 발췌는 답변을 오염시킨다.
- **최소 유사도 임계값**: 이 점수 미만이면 아예 주입하지 않는다. 잡담·무관 질문에 문서를 끼워 넣는 것을 방지한다.

## RAG의 대표적 한계

- **주제 유사 ≠ 정답 포함**: 임베딩 유사도는 "관련된 것"을 찾지 "정답이 있는 것"을 찾지 못한다. 서로 비슷한 문서가 많은 코퍼스(자기유사 코퍼스)에서 특히 판별력이 떨어진다.
- **어휘 불일치**: 질문과 문서가 같은 개념을 다른 단어로 표현하면 놓칠 수 있다.
- **stale 인덱스**: 문서를 고쳐도 재색인 전까지 챗봇은 옛 내용으로 답한다.

## 개선 기법 (난이도 순)

1. 청크 크기/겹침 튜닝, 헤딩 접두어
2. **하이브리드 검색**: 키워드 검색(BM25)과 벡터 검색을 병행해 점수 결합 — 어휘 불일치와 판별력 문제를 동시에 완화
3. **재랭킹(reranking)**: 1차 검색 결과 상위 수십 개를 교차 인코더나 LLM으로 재정렬
4. **쿼리 변환**: 질문을 LLM으로 재작성(HyDE, multi-query)해 검색 친화적으로
