# 밑바닥 구현 vs LangChain — 실측 비교 (Segment 6)

2026-07-11, chatbot-engine의 전기능(스트리밍 대화·메모리·영속화·RAG·knowledge-capture)을 LangChain/LangGraph로 재구현(`experiments/langchain/`)한 실측 비교.

## 코드량

| | 밑바닥 엔진 | LangChain판 |
|---|---|---|
| 기능 코드 | 1,188줄 (TS, 코어 모듈) + 조립/CLI 포함 1,616줄 | **338줄** (JS 4파일) |
| 테스트 | 1,755줄 (114케이스) | 0 (실험 — 실행 AC로 갈음) |
| 런타임 의존성 | **0개** | 7개 (+ 전이 의존성 수백, better-sqlite3 네이티브 빌드 포함) |

약 4~5배 차이. 단, 엔진 1,188줄에는 프레임워크가 영원히 대신해주지 않는 것(도메인 결정들 — 예산 산식, 실패 시 히스토리 미기록, .bak 보존)도 포함되어 있어 순수 "프레임워크가 흡수한 양"은 그보다 작다.

## 대응 관계 (무엇이 무엇을 흡수했나)

| 밑바닥에서 만든 것 | LangChain 대응 | 흡수 정도 |
|---|---|---|
| ndjson 스트리밍 파서 (버퍼링·멀티바이트) | `ChatOllama.stream()` | 완전 — wire 처리가 사라짐 |
| 히스토리 재조립 + 세션 영속화 (JSON, atomic write) | LangGraph state + `SqliteSaver` checkpointer + thread_id | 완전 — 저장/복원 코드가 사라지고 `/clear`도 "새 thread"로 대체 |
| 트리밍+요약 (ContextManager) | 조건부 summarize 노드 + `RemoveMessage` | 구조는 제공, **정책은 여전히 직접** (언제·무엇을 요약할지) |
| 청킹 (헤딩 인지) | `RecursiveCharacterTextSplitter.fromLanguage('markdown')` | 완전 |
| 임베딩 배치 + 벡터 인덱스 + 코사인 | `OllamaEmbeddings` + `MemoryVectorStore` | 검색은 완전, **인덱스 영속화는 미제공** — `memoryVectors` 직렬화를 직접 |
| capture의 JSON 손파싱 (코드펜스 방어 등 40줄) | `withStructuredOutput(zod)` | 완전 — 스키마가 계약이 됨. 단 "빈 결과" 같은 내용 수준 변동은 여전 |
| novelty 판정·저장 정책 | 대응물 없음 — 동일하게 직접 구현 | 0 |

## 실측에서 배운 것 (문서에는 없던 것들)

1. **langchain v1 본체는 에이전트 프레임워크로 얇아졌다.** 벡터스토어·스플리터는 `@langchain/classic`/`@langchain/textsplitters`로 분리, 구 import 경로는 제거. 문서/블로그의 예제 경로가 실물과 다름 — 설치 후 exports 확인이 필수였다.
2. **비교 실험이 원본의 잠복 결함을 드러냈다.** novelty 임계값 0.75가 nomic 임베딩 공간에서 무의미함을 발견(무관한 한국어 문장도 0.77~0.87, 진짜 중복은 0.96+) — 양쪽 모두 0.95로 수정. 원본 AC에서는 캡처 시점 인덱스가 비어 있어 우연히 통과했었다. **같은 기능을 두 번 만들면 가정이 교차 검증된다.**
3. **구조화 출력은 형식 오류를 없애지만 내용 오류는 못 없앤다.** `withStructuredOutput`으로 파싱 실패는 사라졌지만, 8B가 "추출 0건"을 반환하거나 봇의 맞장구("네 기억했습니다")를 지식으로 추출하는 품질 변동은 동일했다.
4. **checkpointer의 대가는 투명성.** 세션이 SQLite 블랙박스에 들어가면서 엔진의 `session.json`처럼 열어보고 고치는 일이 어려워졌다. 디버깅 시 `graph.getState()`를 거쳐야 한다.

## 관점별 평가

- **추상화 이득**: wire/영속화/청킹 같은 "누가 짜도 같은 코드"는 확실히 흡수. 도메인 정책(예산, 임계값, 실패 처리)은 그대로 내 몫.
- **제어권**: 밑바닥판에서 세밀하게 다뤘던 것들(클라이언트 중단 시 히스토리 미기록, 손상 파일 .bak)의 대응물을 프레임워크에서 찾거나 포기해야 함. streamMode 'messages'가 summarize 노드의 토큰까지 흘려보내 노드 필터링이 필요했던 것이 대표 사례.
- **교체 용이성**: ChatOllama → 다른 모델은 한 줄. 우리 엔진도 LlmClient 인터페이스 덕에 동급 — **인터페이스 경계를 직접 설계했다면 프레임워크의 이 이점은 상쇄된다.**
- **디버깅**: 밑바닥판은 스택 전체가 내 코드(원인 판별 즉시). LC판은 문제가 내 코드/프레임워크/버전 조합 중 어디인지부터 갈라야 함 — import 경로 이슈가 그 예고편.

## 결론

원리를 배웠으니 프레임워크의 문서가 아니라 **무엇이 감춰졌는지**가 보인다. 새 프로젝트라면: 표준 부품(스트리밍·청킹·구조화 출력)은 프레임워크, 도메인 정책과 신뢰 경계 처리는 어차피 직접 — 그 경계를 아는 것이 이 프로젝트의 수확.
